MIDI USB bulk endpoint (EP3 OUT) — current understanding
========================================================
Last updated: 2026-04-22
Repo: /home/florian/src-misc/helicon-blender

Goal
----
Make **USB MIDI host→device bulk OUT** (EP 0x03) work on the patched Blender
firmware at High Speed. Device→host (MIDI TX / bulk IN) already works. FS
speed also appears to work in stock firmware.

2026-04-22 runtime delta (new evidence)
---------------------------------------
All findings below are from repeated power-cycle captures using
`firmware/midi_timeout_diag.py --wait-for-power` and UART logs:

- **Patched image is definitely running** (boot tag + handler prints).
- EP3 handle is present and stable:
  - `ep=0002A7E8`, `f=0002140C`, `fn0=0002140C fn4=00021DCC fn8=0001B9E4`
- Host probe now intermittently writes one packet (`WROTE 4`) but most writes
  still timeout.
- Slot-gate fix is active and effective:
  - before: `sl=03>03` (blocked)
  - after:  `sl=00>00` and logs like `gate 03->00 aux 03->00`
- EP flags low-byte fix is active:
  - logs like `fl 00000200->00000202`
- Despite those fixes, receive still does not arm:
  - `pr=00000000` (EP prime reg)
  - `tc=00000000` (trace start callback never called by stock path)
  - no sustained RX progress (`q2` not transitioning to a stable receive flow)

Current interpretation (2026-04-22):
- The failure has moved upstream of qh/slot preparation.
- We can set correct handle pointers/flags and clear slot gates, but firmware
  still does not naturally invoke the EP3 OUT start callback path often enough
  (or at all) to establish steady RX arming.

2026-04-20 direction-label correction (READ THIS FIRST)
-------------------------------------------------------
Previous versions of this doc and `dice_usb_regs.h` had the TCAT direction
labels INVERTED. After decompiling the ISR at `0x21E78` and comparing to
read_start/init code + live traffic on working endpoints, the correct
mapping is:

- **Bank A (`USB_BASE + 0x900`)** = IN bank (device TX, USB IN)
- **Bank B (`USB_BASE + 0xB00`)** = OUT bank (device RX, USB OUT)
- **`0x90000810`** = TCAT_EP_TX_EN (IN direction enable)
- **`0x90000814`** = TCAT_EP_RX_EN (OUT direction enable)
- **`0x90000834`** = IN async prime trigger (NOT OUT as header previously said)
- `TCAT_COMP_EP_IN(n) = 1<<n` (low 16 bits of 0x818) ← correct, IN is low
- `TCAT_COMP_EP_OUT(n) = 1<<(n+16)` (high 16 bits) ← correct, OUT is high

Ghidra's function-name suffixes `_rx` / `_tx` are therefore INVERTED from the
real direction semantics:

- `usb_hw_ep_start_transfer_rx` (`0x21A6C`) → actually the IN submit path
  (calls `dma_start_rx` which pokes the IN-prime register 0x834).
- `usb_hw_ep_start_transfer` (`0x2140C`) → actually the OUT submit path
  (calls `dma_start` which does NOT poke 0x834 — OUT direction has no async
  prime; hardware arms automatically from QH state).
- `usb_hw_ep_dma_start_rx` (`0x21930`) → IN dma_start.
- `usb_hw_ep_dma_start` (`0x20EAC`) → OUT dma_start.
- `usb_hw_ep_rx_enable` (`0x21BA4`) → IN enable (operates on ep+0x24 byte).
- `usb_hw_ep_tx_enable` (`0x21DCC`) → OUT enable (operates on ep+0x20 byte).

The corrected labels are now in `firmware/patch/dice_usb_regs.h` and plate
comments are saved in Ghidra at each of the functions above.

Architecture (what this is NOT)
-------------------------------
- SPI / system DMA (`SPI_BASE` `0xCC000000`, `DMA_BASE` `0x80000000`) from
  `firmware/dice3_hw.h` applies to flash/LED SPI — not USB MIDI.
- USB bulk OUT goes through the USB device controller at `USB_BASE`
  `0x90000000` (ChipIdea-based with TCAT extensions).

Stock firmware call chain (addresses from blender_primary_body.bin, image
base 0x200)
-----------------------------------------------------------------------
- `usb_endpoint_submit_transfer` (`0x2077C`) — fills `ep+0x08` (callback),
  `+0x10` (buf), `+0x14` (size), then calls `ep->start_fn` with no args.
- **OUT path** (correct for EP3 OUT / MIDI RX):
  `usb_hw_ep_start_transfer` (`0x2140C`) → `usb_hw_ep_dma_start` (`0x20EAC`).
  Writes QH[0] |= `0x84000000` (IN_USE + ACTIVE) and QH[4] = size. Does NOT
  poke 0x834 — OUT direction doesn't need it.
- **IN path** (for MIDI TX, EP1/EP2 IN):
  `usb_hw_ep_start_transfer_rx` (`0x21A6C`) → `usb_hw_ep_dma_start_rx`
  (`0x21930`). Writes QH + pokes `0x834` IN-prime bit.
- Endpoint configuration: `usb_endpoint_start_if_configured` (`0x1FF34`)
  dispatches via vtable[2] to `usb_hw_ep_read_start` (`0x2115C`) which
  handles BOTH directions internally (branches on descriptor bit 7). The
  "_read_" name is misleading — it's the unified endpoint-start function.
- Per-EP controller-context array is at the pointer `*(device_ctx + 4)`
  which resolves to `0x2A590` at runtime. Handles for all 12 endpoints
  (6 IN + 6 OUT) live at `0x2A5E8–0x2A867`, 0x40 bytes each. ep_handle IS
  the per-EP controller-slot; they're the same memory region under two
  naming conventions.
- MIDI RX flow: `usb_midi_rx_dispatch` (`0x1B7F0`) →
  `usb_endpoint_submit_transfer` → `ep->start_fn`.
  `usb_midi_rx_start` (`0x1BE50`) calls dispatch only when `ep+0x08` is
  clear AND cables != 0 (re-arm semantics).
- Init function: `usb_hw_ep_table_init` (`0x214A8`) populates all 6 IN
  handles (LOOP 1 with IN-variant handlers + 0x900 bank pointers) and all
  6 OUT handles (LOOP 2 with OUT-variant handlers + 0xB00 bank pointers).

Patch framework current state (`firmware/patch/`)
-------------------------------------------------
- **UART EC1 / EC2 (periodic, ~4 s):** EC1 adds **`dm=`** `TCAT_DMA_CFG` (**`0x90000830`**)
  and **`z6=`** EP0 OUT **QH[6]** (working OUT anchor). EC2 adds **`mps=`** (ep **+0x3C**)
  and **`sl=`** / **`sq=`** — **EP3 controller-slot** bytes **`+0x1F8`>`+0x1FC`** and **QH ptr @
  `+0x1F4`** from `(*(usb_midi_conn+4)) + 2×0x40` (same layout as `usb_hw_ep_read_start` OUT
  branch). Compare **`sl`** gate vs **`ep_handle+0x20`** state in EC1 **`s=`**.
- `boot_init` HS-skip patch at `0x19BE0` = `0xE1530003` (cmp r3,r3 always
  EQ) bypasses the "only arm MIDI at FS" check so that
  `usb_midi_set_interface_handler` runs the MIDI endpoint arming code at
  HS speed. **This is the core patch. It's necessary. It's not sufficient.**
- `midi_loop` hook at `0x5180` replaces `bl midi_rx_poll` with a trampoline
  into `midi_loop_hook` in our SRAM patch zone (0x32600–0x33F80). That
  handler calls `midi_rx_poll` first, then emits periodic EC1/EC2 UART
  diagnostics.
- **REMOVED 2026-04-20**: the one-shot `start_fn` override that forced
  EP3 OUT's `start_fn` to `usb_hw_ep_start_transfer_rx` (0x21A6C). Under
  the corrected direction convention, 0x21A6C is the IN path; forcing it
  on an OUT endpoint routes through dma_start_rx which pokes 0x834
  (IN-prime, no effect on OUT direction) and skips the proper OUT arming
  mechanism. Stock init's `start_fn = 0x2140C` is correct for OUT.
- **REMOVED 2026-04-20**: the periodic `TCAT_EP_ASYNC_PRIME |= EP_BIT_RX(3)`
  reinforcement poke. 0x834 is IN-prime; EP_BIT_RX(3) = bit 3 primes
  EP3 IN not OUT. Was a no-op for MIDI RX.

Live runtime state (captured 2026-04-20 on patched fw)
------------------------------------------------------
EP3 OUT ep_handle @ `0x2A7E8`:
```
  +0x00 start_fn     = 0x0002140C   (stock OUT submit — correct)
  +0x04 set_halted   = 0x00021DCC   (stock OUT enable — correct)
  +0x08 complete_fn  = 0x0001B9E4   (usb_midi_rx_complete)
  +0x10 buffer       = 0x00030D64
  +0x14 size         = 0x00000200
  +0x1C qh_ptr       = 0x90000B60   (OUT bank EP3 — correct)
  +0x20 state        = 0x00000003   (DMA_ACTIVE)
  +0x24 flags        = 0x00000203
```

OUT bank EP3 QH @ `0x90000B60`:
```
  q0 = 0x80088200   (IN_USE + BULK + bit 15 + MPS=0x200)
  q2 = 0x00000000   (no dTD overlay — hardware never populated it)
  q4 = 0x00080200   (transfer size valid + size=0x200)
  q6 = 0x00000010   (idle — NOT armed)
```

For comparison, WORKING EP0 OUT QH @ `0x90000B00`:
```
  q0 = 0x80028000   (IN_USE + bit 17 + bit 15)
  q2 = 0x0000A010   (dTD overlay populated)
  q4 = 0x40000040
  q6 = 0x00000080   (ACTIVE bit 7 — armed)
```

The critical difference: EP0 OUT has q2 populated and q6 at 0x80 (hardware
armed). EP3 OUT has q2=0 and q6=0x10 (idle). Software side completed its
part (dma_start ran, wrote q0 IN_USE and q4 size) but the TCAT hardware
never transitioned the QH into armed state. Without q6=0x80 and a
populated dTD overlay, host OUT tokens are NAK'd → pyusb TIMEOUT.

Wire-level symptoms
-------------------
- `pyusb` writes to EP 0x03 return TIMEOUT consistently at HS.
- No `USBERR I/O` (STALL) — the endpoint accepts tokens but NAKs them.
- Device-side UART (`ptr_midi_streaming_state+0x1c` byte = 0x02 = HS per
  the speed code; other evidence suggests the "== 2" arm condition was
  actually the FS condition, and at HS this byte has a different value
  that bypasses MIDI arming — which is why the 0x19BE0 patch is needed).

Root-cause narrowing (2026-04-20)
---------------------------------
Confirmed NOT the bug:
- qh_ptr bank selection (correctly 0x90000B60 = OUT bank under corrected labels).
- start_fn routing (correctly 0x2140C = OUT submit).
- set_halted_fn (correctly 0x21DCC = OUT enable).
- TCAT_EP_RX_EN bit 3 — correctly set at 0x814 (=0x2F has bit 3).
- TCAT_EP_COMP_ENABLE bit 19 — correctly set at 0x81C (=0x90007 has bit 19).
- dma_start bit-26 (ACTIVE) set in QH[0] — at one point, but hw clears
  it without firing completion.

Confirmed the gap: **hardware does not populate q2 / transition q6 from
0x10 → 0x80 for EP3 OUT.** Something in the HS-arming path that stock
firmware skips (hence the 0x19BE0 bypass bug) is incomplete — bypassing
the check lets later code run, but something else is still missing or
misconfigured that would transition the OUT QH to hardware-armed state.

Root-cause narrowing addendum (2026-04-22)
------------------------------------------
Additional hypotheses tested with runtime evidence:

- Rejected: wrong EP handle function pointer (`f`) for EP3 OUT.
  Runtime repeatedly shows `f=0002140C` and boot one-shot logs show
  `fn0=0002140C fn4=00021DCC fn8=0001B9E4`.
- Rejected: slot gate stuck nonzero as sole blocker.
  Runtime now shows `sl=00>00` after explicit clear path.
- Rejected: flags low-byte mismatch as sole blocker.
  Runtime logs show repeated `fl 00000200->00000202`.
- Still confirmed: start trace callback is not invoked by normal flow:
  `tc=00000000` across captures, even with EP handle/gate/flags corrected.

Working conclusion:
- HS MIDI RX path is failing at a control-flow stage before (or bypassing)
  regular EP3 start callback invocation. Current best next step is to trace
  where and why EP3 scheduling is suppressed after `usb_midi_rx_dispatch`.

Suggested next-round investigation
----------------------------------
1. **TCAT reg `0x90000830` (DMA cfg)** — documented live value 0x0C100020 in
   `firmware/dice3_usb_registers.md`. **`blender_primary_body.bin` has no
   data-pool literal `0x0C100020` and no instruction uses offset `#0x830` from
   `0x90000000` in a simple listing pass** — so this firmware image may never
   poke it (boot ROM / one-shot init elsewhere, or table in another segment).
   **`TCAT_DMA_CFG`** is now declared in `firmware/patch/dice_usb_regs.h` for
   on-device dumps and UART probes. Next: compare **cold boot vs configured
   HS** and **FS vs HS** after `usb_midi_set_interface_handler`.
2. **Walk FS enumeration path vs HS.** FS works pre-patch. Find the
   divergence: what does FS path do at submit-time that HS doesn't?
   Specifically, after usb_midi_rx_start → dispatch → submit_transfer →
   dma_start, what transitions q6 to 0x80?
3. **Check dTD allocation / next-dTD pointer.** In ChipIdea, QH+0x18 (q6)
   can hold a next-dTD pointer. EP0 OUT has q6=0x80 (bit 7 — might be
   ACTIVE or a flag on the pointer). Maybe MIDI OUT needs a dTD to be
   allocated and linked; stock TCAT driver might be missing this for
   bulk OUT endpoints at HS.
4. **`usb_hw_ep_read_start` (`0x2115C`) OUT slot state @ +0x1F8** — Decompilation
   shows the OUT descriptor path only enters if **`(slot + 0x1F8) == 0`**; on
   success it stores **2** there (not 3). The **`ep_handle +0x20` "state=3
   (DMA_ACTIVE)"** in UART dumps is the **per-endpoint handle** field — do not
   conflate with **`slot+0x1F8`** without mapping `0x2A7E8` → controller slot
   base for EP3 OUT. If **`read_start` never ran** for MIDI EP at HS, QH caps
   stay half-programmed relative to a control OUT path.
5. **ISR OUT completion loop (`usb_hw_dsr_interrupt_dispatch` ~`0x22320`–`0x225D4`)** —
   For each bit in the OUT completion mask, the ISR walks **dTD/QH state at
   `slot+0x1F4`**, may call work that **re-sets QH[0] `| 0x84000000`** and
   `usb_hw_ep_dma_continue`-like logic. If **EP3 never raises completion**, this
   loop never feeds the hardware path that EP0 OUT sees — consistent with **q2=0 /
   q6 stuck at idle** on EP3 OUT.
6. **Breakpoint trace.** Set a software bp at 0x2140C (start_transfer
   entry) to confirm it's actually called for EP3 OUT. And at 0x20EAC
   (dma_start) to confirm IT's called with ep=EP3_OUT handle. And at
   0x2115C (read_start) to confirm IT ran its OUT branch to completion.
   Caveat: JTAG halt disrupts DMA — can only breakpoint at idle, and
   replug device after each halt cycle.

Operational notes
-----------------
- Flash: `python3 firmware/jtag_flash_v2.py --mode flash-all --ref
  firmware/blender_spi_patched.bin`. ~48s. Reported "FAILED ops" are
  golden-protection-zone refusals and block→sector fallbacks — expected.
- After flash: **power-cycle the device** (not just JTAG reboot — reboot
  stub was flaky in this session).
- JTAG halt corrupts in-flight USB/DMA; read state between host-traffic
  attempts, not during.
- Device is self-powered; USB replug doesn't reset SRAM. A real
  power-cycle (unplug power) does reset SRAM.
